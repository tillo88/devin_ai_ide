## Obiettivo generale

DEVIN AI IDE è un coding agent locale, desktop-first, simile a Codex/Claude Desktop/Kimi Work. The best they have all togheter if possible.

Architettura principale:

* backend FastAPI;
* interfaccia moderna `/app`;
* shell desktop Tauri;
* chat e contesto per progetto;
* orchestratore `Planner → Coder → Patcher → Runner → Critic`;
* sandbox sicura con diff e applicazione controllata;
* routing tra modelli locali e rig esterno;
* memoria anti-contaminazione;
* training/eval con quality gate deterministico e revisione umana/Teacher.

Non devi reinventare il progetto né ripartire da zero: devi prima comprenderne lo stato reale, verificare cosa è già implementato e continuare dal punto più avanzato.

## Regole di sicurezza e qualità

### File da non leggere, stampare o committare

Non accedere inutilmente e non mostrare il contenuto di:

```text
.env
tinyfish api.txt
chiavi API
token
credenziali
local_memories.jsonl
log runtime
modelli GGUF
file pickle/cache
workspace runtime dell’utente
```

Non aggiungere segreti a output, patch, test o commit.

### Memoria anti-contaminazione

Non promuovere automaticamente risultati di benchmark o ipotesi nella memoria recall-safe.

Solo questi elementi possono diventare conoscenza affidabile:

* risultati verificati;
* correzioni confermate;
* lezioni approvate;
* fallimenti con causa, evidenza e regola di retry.

### Modifiche

* preferisci patch piccole e leggibili;
* evita riscritture complete di file grandi;
* mantieni compatibilità con il comportamento esistente;
* aggiungi un regression test per ogni bug corretto;
* non “aggiustare” test corretti solo per farli passare;
* non mascherare errori con `except Exception: pass`;
* non introdurre fallback che dichiarano successo senza verifica;
* evita dipendenze cloud quando esiste un’alternativa locale ragionevole.

### Sintassi e generazione file

Controlla con particolare attenzione:

* f-string con parentesi o virgolette annidate;
* stringhe multilinea generate;
* escaping Windows/WSL;
* path con spazi;
* comandi PowerShell;
* heredoc;
* newline inserite da generatori;
* JavaScript generato dentro template Python;
* JSON scritto tramite concatenazione manuale.

Per contenuti complessi generati da script, preferisci serializzazione strutturata o `repr()` invece di quoting annidato fragile.

## Gestione dei bug risolti

Quando correggi un bug:

1. riproducilo;
2. identifica la causa radice;
3. applica la correzione minima;
4. crea un regression test;
5. esegui test mirati e suite completa;
6. aggiungi una breve nota al file `docs/CONTINUITY_*` più recente, oppure crea un nuovo file datato se la sessione è successiva;
7. se dal bug deriva una regola generale per le future AI, aggiorna anche `AGENTS.md`;
8. indica chiaramente:

   * causa;
   * file modificati;
   * test eseguiti;
   * risultato.

Non segnare un problema come risolto basandoti solo sulla lettura del codice.
